Izanami allow to define finely when a feature should or should not be active, but sometimes this is not enough?

For instance you may want to have different activation strategy for dev and prod environnement.

There is two way to make it work, however they both have their drawback :

- You could use a user list strategy and pass your environnement as a user. This is a bit hacky, but would work as long as you don't need to add another activation strategy to this feature.
- You could use a script strategy, and pass your environnement as payload for your feature activation request. This works, however it's better to avoid using script feature when possible, since this kind of feature strategy is impossible to cache client side.

Since this kind of use case is very common, Izanami provide **contexts**.

**Contexts** are a way for callers to indicate their execution context. They are organised as a tree : one context can define subcontexts, which can themself have subcontexts and so on.

<Note>
  Context are ideal to let caller indicate information such as browser, env,
  device, ...
</Note>

You can **redefine activation strategies** for your feature for a given context.

Caller can specify a context or subcontext when checking for feature activation status.

When called with a context, Izanami will search for context-specific activation startegies and use them if they exist.
If they don't, Izanami will go up this _context tree_ and use the first encountered activation strategy, or the base one if no strategy was encountered.

The above schema represents a "context tree". The base node of the tree represents strategies that is defined by feature.
Each node may or may not redefine strategy for these features. To keep it simple, we only used "basic" startegy (feature is either enable or disable).

Now let's see what happen when requesting feature activation :

- feature1 without specifying context : <span style={{color: "green", fontWeight: "bold"}}>enabled</span> (easy, base strategy was used here)
- feature1 for context1 : <span style={{color: "red", fontWeight: "bold"}}>disabled</span> (we use context1 redefinition for feature1)
- feature1 for subcontext2 : <span style={{color: "red", fontWeight: "bold"}}>disabled</span> (there is no activation strategy redefinition for subcontext2, therefore we use the closest one when going up : context1 redefinition)
- feature2 for subcontext1 : <span style={{color: "red", fontWeight: "bold"}}>disabled</span> (context2 has no overload for subcontext1 nor context1, therefore base activation strategy is used)
