In Izanami, feature flags are just called "features" for the sake of simplicity.

## Activation strategy

Activation strategies are used to define finely when a feature should be active.

You can define your activation strategy using below conditions.

A strategy can combine multiple condition, in such case **all conditions must be active** to activate the feature.

Additionlly, one feature can have several strategies, **only one strategy need to be active** to activate the feature.

To sum up, there is a **logical OR between strategies** (if many) and a **logical AND between one strategy's conditions**.

### Default strategy

The default strategy is the one you get when creating a feature.

If feature is enabled, it's considered active, otherwise it's not.

### Time based strategies

With this kind of feature, Izanami will activate feature when current time is in provided range.

These startegies are used to implement <a href="" target="_blank">release</a> or <a href="" target="_blank">ops</a> flags.

There is several variation for this catrgory of feature, that can be composed to craft exactly what you need.

- Date range feature : define a start and/or end date for feature activation
- Hour range : feature will be active every day on provided range(s)
- Activation days : feature will be active only on provided days

### User based strategies

With user based strategies, activation conditions are based on <a href="#" target="_blank">provided user</a>.

Three is two types of user based strategies :

- user list : define a list of user for which feature will be active. This strategy is used to implement <a href="" target="_blank">permission flag</a>.
- percentage : feature will be active only for givent percentage of users. Percentage is based on user hash (using murmum3), this means that activation is consistent for a given user (if feature is active once, it'll remain active). This strategy is used to implement <a href="" target="_blank">canary flag</a>.

### Script strategies

Sometimes above strategies are not enough. For instance feature activation decision may depend on more than "just" the user.

For such case, Izanami offers (with the help of [WASMO](https://github.com/MAIF/wasmo)) the possibilty to define "script feature".

Such features rely on an underlying wasm script, which will be executed with provided input each time feature activation is required.

To learn more about these script, refer to the <a href="#" target="_blank">wasm script section</a>.
